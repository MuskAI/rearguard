# Explicit Watermark Ensemble

Detection-only baseline for visible watermark evidence. It does not remove or
rewrite image pixels.

## Pipeline

1. The existing `yolo-watermark` service localizes candidate regions.
2. `remove-ai-watermarks` keeps the high-precision known-provider registry.
3. RapidOCR reads text inside candidate regions.
4. A persistent FAISS gallery retrieves candidates using CLIP embeddings of a
   `bright_overlay_v1` view. That view suppresses scene content and preserves
   pale, locally brighter watermark strokes.
5. OCR wording is interpreted separately as `supports_ai_generation`,
   `inconclusive`, or `not_supported`; generic text is not treated as an AI
   generation claim.
6. Conservative fusion returns `text`, `logo`, `unknown`, or `none`.

OCR-only text is never promoted to a watermark without a detector or registry
candidate. Retrieval attribution must pass a calibrated per-platform threshold,
an inter-platform margin, and position validation. OCR may resolve a close
platform tie only when that platform's own retrieval score and position still
pass. A failed lookup remains `unknown` instead of being forced into a platform.

The `explicitWatermark.aiGenerationTextSignal` field contains the OCR wording,
matched AI/platform terms, `likelyAIgenerated`, and a separate
`aiGenerationConfidence`. This is supporting watermark evidence, not a claim
that arbitrary image text proves an image was generated by AI.

## Internal API

`POST http://127.0.0.1:5068/v1/analyze` accepts a multipart `file`, plus JSON
form fields `candidates` and `registryHits`. The response contains:

```json
{
  "explicitWatermark": {
    "detected": true,
    "type": "text|logo|unknown|none",
    "sourcePlatform": "豆包",
    "confidence": 0.88,
    "hits": [{
      "bbox": {"x": 0.8, "y": 0.9, "w": 0.15, "h": 0.06},
      "type": "text",
      "text": "...",
      "sourcePlatform": "豆包",
      "retrievalSimilarity": 0.0,
      "retrievalThreshold": 0.8554,
      "retrievalMargin": 0.05,
      "retrievalAccepted": true,
      "retrievalReferenceId": "doubao-black",
      "retrievalReason": "accepted",
      "position": "bottom-right",
      "areaRatio": 0.009,
      "opacityEstimate": 0.5,
      "confidence": 0.88
    }]
  }
}
```

`bbox` uses `display_normalized_v1` coordinates. `opacityEstimate` is a visual
translucency proxy, not an alpha-channel measurement.

## 66 deployment

- Fusion service: `realguard-watermark-ensemble.service`, `127.0.0.1:5068`
- Existing precheck: `realguard-watermark-precheck.service`, `127.0.0.1:5066`
- Existing localization model: `realguard-yolo-watermark.service`, `127.0.0.1:5067`
- Existing main detector: `realguard-detection.service`, `127.0.0.1:5000`

The 5066 adapter calls 5068 after the registry and YOLO branches finish. The
main `/image` route also forwards `data.explicitWatermark` and the full
`data.watermarkPrecheck` payload. If OCR, gallery retrieval, or 5068 is
unavailable, the main detector continues and reports a degraded watermark
branch instead of failing the image detection request.

## Retrieval index and tuning

The checked-in `retrieval/` directory contains a 20-entry `IndexFlatIP` FAISS
index, source metadata, reference crops, and platform calibration. Sources are
public captures from `wiltodelta/remove-ai-watermarks`. The current calibration
uses 69 negative corner crops. Gemini remains intentionally conservative because
its positive and negative calibration bands overlap.

Rebuild on a host that already has the configured CLIP model cached:

```bash
.venv/bin/python build_retrieval_index.py \
  --data-root /path/to/remove-ai-watermarks/data \
  --negative-root /path/to/camera-negatives \
  --negative-root /path/to/cleaned-negatives \
  --output-dir ./retrieval \
  --device cpu
```

`WATERMARK_CLIP_DEVICE=auto` checks the GPU compute capability against the
architectures compiled into PyTorch. It falls back to CPU when they do not
match, as on the TITAN Xp host with the current PyTorch build.

Important settings are `WATERMARK_OCR_THRESHOLD`,
`WATERMARK_RETRIEVAL_BACKEND=faiss_clip`,
`WATERMARK_RETRIEVAL_INDEX_DIR`, and `WATERMARK_EVIDENCE_WORKERS`.
