# 慧鉴 AI 公网入口压测报告

## 1. 执行摘要

- 测试时间：2026-07-18 18:42-18:43（Asia/Shanghai）
- 测试目标：`https://www.rrreal.cn/`
- 测试范围：公网 HTTPS 首页、单一外部来源 IP、无登录、无文件上传、无模型推理
- 结论：在并发 10、总请求 100 的场景下完成 100%，吞吐 73.73 req/s，P95 195 ms；并发提升到 20、总请求 500 时，Nginx 主动返回 345 个 429，说明入口限流按设计生效。
- 边界：本报告验证的是“单 IP 公网入口安全容量”，不是 GPU 推理链路的最大吞吐。不能据此宣称系统支持 73.73 次图像检测/秒。

## 2. 测试目标

本轮测试回答三个问题：

1. 网站在温和并发下是否稳定响应。
2. 突发流量是否会触发入口保护，并在压力结束后恢复。
3. 压测是否影响主站、鉴伪后端和 GPU 模型服务的存活状态。

本轮不测试登录、上传、快速检测、Swarm 检测、报告生成、数据库写入和 GPU 多任务并发，这些应在专用测试账户和白名单压测机上单独执行。

## 3. 环境与方法

| 项目 | 内容 |
|---|---|
| 入口 | `https://www.rrreal.cn/` |
| 协议 | HTTPS |
| 流量来源 | 单一外部来源 IP |
| 请求 | `GET /` |
| 工具特征 | ApacheBench User-Agent |
| 观测 | 客户端延迟、HTTP 状态、Nginx access log、systemd 服务、模型健康接口 |
| 风险控制 | 由低到高分级加压；不绕过 Nginx 限流；结束后立即执行健康检查 |

基线探测返回 HTTP 200，耗时 66.261 ms，响应体 1,188 字节。

## 5. 限流行为解释

线上 Nginx 当前配置了分层限流：

| 入口 | 速率 | 突发 |
|---|---:|---:|
| 通用页面 | 20 req/s | 80，`nodelay` |
| API | 8 req/s | 40，`nodelay` |
| 上传 | 30 req/min | 20，`nodelay` |

并发 20 的 500 请求场景超过单 IP 通用页面令牌桶，Nginx access log 明确记录 429。该现象是入口保护动作，不是应用崩溃。它能限制单来源突发流量，但也意味着 NAT、校园网或企业出口下的大量正常用户可能共享一个公网 IP，需要结合真实用户规模持续观察误伤率。

## 6. 压测后健康检查

压测结束后立即检查：

- 首页：HTTP 200，约 59 ms。
- `realguard-backend.service`：`active`。
- `realguard-detector-backend.service`：`active`。
- 模型健康接口：`status=ok`，远程 CUDA 就绪，观测延迟 23.83 ms。

本轮未发现压测导致的服务退出或持续不可用。

## 7. 容量判断

### 已验证

- 单 IP、首页轻请求、并发 10 时可稳定完成 100 个请求。
- 单 IP 突发超过入口策略时会被 429 抑制。
- 压力结束后主站和鉴伪后端仍健康。

### 尚未验证

- 多 IP 用户的公网聚合吞吐。
- 登录态请求和数据库连接池上限。
- 图片上传带宽、请求体限制和临时文件 I/O。
- 快速检测与 Swarm 检测的 GPU 并发、显存峰值和排队时间。
- PDF/图片报告生成的 CPU、内存和文件系统压力。
- 10 分钟以上稳态压力、峰值恢复时间和内存泄漏。

因此，当前可安全表述为：“公网入口在单 IP 并发 10 下稳定，过载由 Nginx 429 保护。”尚不能给出整个鉴伪系统的最终极限检测 TPS。

## 8. 下一轮正式容量测试

建议在业务低峰建立一台白名单压测机，使用专用测试账号和固定测试图，分四条链路执行：

| 阶段 | 链路 | 建议并发 | 时长 | 重点指标 |
|---|---|---:|---:|---|
| A | 登录与历史记录只读 | 1/5/10/20 | 每档 5 min | P95、错误率、DB 连接数 |
| B | 图片上传但不推理 | 1/2/5/10 | 每档 5 min | 带宽、临时盘、413/429 |
| C | 快速检测 | 1/2/4/8 | 每档 10 min | GPU 利用率、显存、队列、P95 |
| D | Swarm 检测 | 1/2/3/4 | 每档 10 min | 子任务扇出、总耗时、超时率 |

统一停止条件：错误率连续 1 分钟高于 2%；P95 连续 3 分钟超过业务目标两倍；GPU OOM；主机内存超过 90%；磁盘可用空间低于 15%；数据库连接池耗尽；任一核心 systemd 服务重启。

## 9. 改进建议

1. 将入口限流指标、429 比例、模型队列长度和 GPU 显存纳入统一监控。
2. 对登录用户按账号或 API key 限流，避免仅按 NAT 出口 IP 判断。
3. 上传与推理使用异步任务队列，向前端返回排队位置和预计等待时间。
4. 快速检测与 Swarm 分开设置并发池和超时，防止慢任务占满全部工作进程。
5. 正式极限测试保留原始测试命令、请求样本、服务日志和 GPU 时序，生成可复现实验附件。
